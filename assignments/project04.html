Due: Mon, Nov 6, 2023 at 11:59 PM to Github Classroom Assignment

Requirements

  1. You will evolve your project03 code to use a persistent SQLite database rather than an in-memory map of maps
  2. Your solution must continue to support recent feature additions, such as concurrency and robots.txt
  3. Your solution must protect your database from concurrent writes
  4. Your solution must not introduce SQL Injection vulnerabilities

Given

  1. We will discuss several approaches to program SQLite databases using Go
  2. We will use the same TF-IDF test cases from lab05 (that is, disallowing chap31.html) and corpus from project03
  3. In order to complete crawling in less than 30 seconds as required by go test, a new robots.txt is given, which omits crawl-delay

Rubric

  1. 10 pts. TestDisallow (same results as the in-memory map of maps)
  2. 50 pts. TestTfIdf
  3. 40 pts. Code review for SQL and relational design. Please pass the test cases before coming to the code review meeting.
  4. 5 pts. (Extra credit)
    1. Use a Go interface to build an abstraction for the persistent SQLite database vs. the in-memory map of maps
    2. Make the indexing code switchable on the command line using a Go flag, e.g. 
       $ go run . -index=inmem

      or

       $ go run . -index=sqlite

Implementation tips

  1. You’ll need SQL pragma foreign_keys = on; for each database connection.
  2. If you have SQL UNIQUE columns, you don’t need to add an index for those because SQLite does so automatically.
  3. However, if you want to SELECT for columns in a table, SQLite needs to have an index for each searchable column. In lecture, we described that the hits (or frequency) table contains non-unique columns that you might want to SELECT for
  4. If you want to search the SQL index using a join, here’s a tip for how to start that query SELECT urls.name, hits.count FROM urls. You can finish it.
